# Registratura тестовое задание

## Задачка у нас следующая: 
- Мы хотим реализовать сервис, который выполняет следующую бизнес фукнцию: 
    - “Клиент заходит в личный кабинет сайта и пополняет баланс своего кошелька”
    - “Процессинг оплаты происходит через некоего “платежного оператора”” (имитируем этот сервис локальным докер контейнером который на вход принимает REST запрос и на выходе по результату обработки стучится в наш webhook)

## Ограничения и особенности: 
- Хотим Фронт и Бэк завернуть в докер образы, мб docker compose, чтобы локально все запускалось по одному клику (или с минимальным кол-вом кликов) и понятной документацией
- Проект хотим сделать наиболее компактным - с минимальным количеством строк кода. Ваш код будет ревьювить человек, не агент :)) 
- Хотим особое внимание уделить целостности платежей, валидациям платежных данных, валдиациям денег, хранениям денежных операций. 
- Фронт достаточно минималистичный (простая форма суммы, кнопка “Пополнить, список последних платежей, текущий баланс, можно без авторизации (один пользователь на время запуска проекта, при перезапуске докера инфа затирается) 
- Полезные ключевые слова, которые могут помочь в реализации финансовой части: Ledger, Finite State Machine 

## Список ожидаемых docker образов: 
- Фронт на React
- Backend: Django, DRF
- "Внежний платежный сервис" - отдельный “внешний сервис” - имитация Платежного Оператора (REST + webhook отработка) 
- Очередь (Rabbit MQ) между Back и Воркером на отправку платежа
- Postgres DB
- REST на бэке для приема результатов платежа (то есть обработка webhook). 

## Комментарий ревьювера: 
- ! Можно использовать любые агентские/LLM системы, 
- ! Ожидаем полного погружения в созданное решение и его защиту перед ревьювером. 
- Я как пользователь системы хочу развернуть локально докер образы и суметь пройти клиентский путь. 
- Ролевую модель можно очень базовую, либо без авторизации
- Система пусть генерирует случайные тестовые данные (либо 3-5 тестовых наборов данных захардкодить и случайно подставлять в запросы)
- Что оцениваем: корректность денежной модели, защиту от дублей, атомарность платежа, понятность кода, компактность решения, readme и локальный запуск, тесты на ключевые сценарии. 
- Что не оцениваем: UI и его красоту, авторизацию. 
- Результат ожидаем в публичном репозитории github/gitlab
